# SE2 Handoff — April 9, 2026

**From:** Tyrone Yazon (customer module)  
**To:** SE2 staff POS team  
**Re:** Changes made today that are consequential for your side

This document covers only the April 9 changes that have a direct impact on SE2's integration with the shared database. Frontend-only changes to the customer app are excluded.

---

## 1. Legacy `place_customer_order` RPC dropped

**What changed:** The 3-argument version of `place_customer_order(uuid, varchar, varchar)` has been dropped from the database.

**Impact on SE2:** None — your POS uses direct `INSERT` into `orders`, not this RPC. The 4-argument customer version (`place_customer_order(uuid, text, text, jsonb)`) is unchanged and still active.

**Why it was dropped:** The 3-arg version was dead code that could be called by the `anon` role to create orders without any items attached. It was an exploitable orphan-order vector.

---

## 2. EXECUTE grants on customer RPCs tightened to `anon` only

**What changed:** The following RPCs have had EXECUTE revoked from `PUBLIC` and `authenticated`, and are now granted exclusively to `anon`:

- `place_customer_order(uuid, text, text, jsonb)`
- `cancel_customer_order(bigint, uuid)`
- `fetch_customer_orders(uuid)`
- `fetch_customer_order_by_id(bigint, uuid)`
- `fetch_customer_order_items(bigint, uuid)`
- `verify_guest_session(uuid)`
- `get_best_sellers(integer)`

**Impact on SE2:** None — your POS uses direct table access (`SELECT`, `UPDATE`, etc. on `orders`, `order_items`, etc.) and does not call these RPCs. The `authenticated` role's table permissions are completely unchanged.

**Why it was changed:** Defense-in-depth. The `authenticated` role (your staff role) has no business calling customer session-scoped RPCs. Revoking prevents accidental or malicious misuse.

**If SE2 ever needs to call one of these RPCs:** Reach out and we can add `authenticated` back for the specific function needed.

---

## 3. Advisory lock added to `generate_queue_number`

**What changed:** `generate_queue_number()` now acquires a per-day transaction-level advisory lock (`pg_advisory_xact_lock`) before counting existing queue numbers for the day.

**Impact on SE2:** None — queue numbers are only generated by the `set_queue_number` trigger on customer orders (where `sessionID IS NOT NULL`). Staff-created orders (where `sessionID IS NULL`) do not trigger queue number generation. The lock is held only for the duration of the INSERT transaction.

**Why it was changed:** Pre1vents the rare race condition where two simultaneous customer orders could receive the same queue number.

---

## 4. `place_customer_order` now validates quantity (1–99 per item)

**What changed:** The 4-arg `place_customer_order` RPC now rejects items with `quantity < 1` or `quantity > 99`, and rejects empty item arrays. Returns structured JSON errors (`'invalid_quantity'`, `'empty_order'`) instead of inserting bad data.

**Impact on SE2:** None — your POS inserts directly into `orders` and `order_items`, not through this RPC.

---

## 5. `Ready` status — dropped from customer app

**What changed:** After SE2 confirmed that `Ready` is not used as a distinct step in the POS workflow (orders go directly `Preparing → Completed`), the customer app has removed all `Ready`-specific handling. The `Ready` branch has been removed from the status badge, polling logic, order card styling, and active-status filter.

**Impact on SE2:** None — the `chk_orders_status` DB constraint still allows `'Ready'` at the database level. If SE2 ever sets an order to `Ready`, the customer app will treat it like any other non-terminal status (badge shows the generic Pending style). No customer-facing crash or error.

**If SE2 decides to use `Ready` in the future:** Notify the customer module developer — re-adding support is straightforward.

---

## No changes to

- `orders`, `order_items`, `products`, `addons`, `guest_sessions` table schemas
- Any RLS policies
- `authenticated` role permissions
- SE2-owned RPCs (`get_my_role`, `get_session_id`)
- `users` or `payments` tables
- Any triggers other than the internal implementation of `generate_queue_number`
