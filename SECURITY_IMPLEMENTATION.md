# Goat2Go Customer Module — Security Implementation

## Overview

This document covers all security logic implemented in the customer ordering module and its Supabase backend. It is intended as a reference for the developer and the staff/admin team.

**What the security model protects against:**
- Anonymous users reading other customers' orders
- Spam/abuse of the order placement endpoint
- Accidental or malicious exposure of staff/payment data to customers
- Information leakage through error messages

**Two-role model:**
- `anon` — unauthenticated customers. Minimal permissions, enforced by RLS.
- `authenticated` — staff and admin. Full CRUD on all operational tables.

**Core principle:** The `anon` role can only INSERT orders and SELECT menu data. All sensitive reads go through `SECURITY DEFINER` RPC functions that verify session ownership internally before returning any data.

---

## Part A — Order Security

### A1. Session Management

Every customer gets an anonymous session UUID that scopes all their order activity.

**How it works (`client/src/contexts/GuestSessionContext.tsx`):**
1. On app load, reads `localStorage.getItem('goat2go_session_id')`
2. Calls `verify_guest_session(p_session_id)` RPC to confirm the session still exists in the DB
3. If session is missing or invalid: generates a new UUID via `crypto.randomUUID()`, inserts into `guest_sessions`, and saves to localStorage
4. Sessions expire server-side via `expires_at` (24 hours from creation). `verify_guest_session` enforces this with `AND expires_at > now()` — expired sessions return null and the frontend creates a new one. Previously this was not enforced, allowing old sessions and their orders to persist indefinitely.

**Key design decisions:**
- Session ID is always passed explicitly as a parameter to every order RPC — the server never infers it from cookies or headers
- The `anon` role has INSERT-only on `guest_sessions` — customers cannot read or modify other sessions
- Session persistence survives page reload but is verified against the DB on every load

### A2. Order Placement & Rate Limiting

**RPC: `place_customer_order(p_session_id, p_order_type, p_discount_type, p_items)` — SECURITY DEFINER**

Server-side validation enforced before any order is created, in this order:

1. **Empty-payload rejection** — if `p_items` is NULL, not an array, or an empty array, returns `{success: false, error: 'empty_order'}` immediately.
2. **Quantity bounds** — every item must have `quantity` between 1 and 99 inclusive. Rejects with `{success: false, error: 'invalid_quantity'}` before any DB write. Prevents negative-quantity money exploits.
3. **Rate limit** — maximum 5 orders per session per 10 minutes. Returns `{success: false, error: 'rate_limit_exceeded'}`.

```sql
-- Quantity validation (runs before rate limit and before any INSERT)
FOR v_item IN SELECT * FROM jsonb_array_elements(p_items) LOOP
  v_qty := COALESCE((v_item->>'quantity')::INT, 0);
  IF v_qty < 1 OR v_qty > 99 THEN
    RETURN jsonb_build_object('success', false, 'error', 'invalid_quantity', ...);
  END IF;
END LOOP;
```

- All validation is entirely server-side — cannot be bypassed from the client
- Returns structured JSON errors instead of raising Postgres exceptions (friendlier for the frontend to handle)

**Server-side price validation:**

The RPC ignores the `price` field sent by the client for each item. Instead, it fetches the real price directly from the `products` table (`WHERE "productID" = ... AND is_current = true AND status = 'AVAILABLE'`) at the moment of order creation. The same applies to add-on prices — each add-on's price is looked up from the `addons` table (`WHERE id = ... AND is_available = true`) and a clean JSONB array is rebuilt server-side before being written to `order_items.selectedAddons`. A client that intercepts the network request and modifies any price field will still have the real price stored.

- `order_items.price` is always the current server price at order time
- `order_items.selectedAddons[].price` values are always server-sourced
- The frontend payload contract is unchanged — the client still sends `price` for its own display logic; the RPC discards it

**What the RPC sets on the new order:**
- `status = 'Pending'`, `paymentstatus = 'Unpaid'` (set by DB defaults)
- `sessionID`, `orderType`, `discountType` from parameters

**What it never sets:**
- `userID`, `completeTimestamp` — staff-only fields, never written by the customer app

**Queue number:** Auto-generated by the `set_queue_number` trigger (INVOKER). The trigger only fires when `sessionID IS NOT NULL`, meaning it has no effect on staff-created orders.

### A3. Order Read Security (Session Scoping)

The `anon` role has **zero SELECT** on `orders`, `order_items`, and `guest_sessions`. All customer reads go through SECURITY DEFINER RPCs that enforce session ownership:

| RPC | Scoping Logic |
|---|---|
| `fetch_customer_orders(p_session_id)` | `WHERE "sessionID" = p_session_id AND "orderTimestamp" >= now() - interval '24 hours'` — limits to last 24 hours so expired session orders don't bleed through |
| `fetch_customer_order_by_id(p_order_id, p_session_id)` | `WHERE "orderID" = p_order_id AND "sessionID" = p_session_id` — returns `NULL` if either doesn't match |
| `fetch_customer_order_items(p_order_id, p_session_id)` | Checks parent order belongs to session first — returns empty set if not |
| `verify_guest_session(p_session_id)` | `WHERE id = p_session_id AND expires_at > now()` — returns `NULL` if not found **or expired**. Frontend creates a new session on null. |

A customer cannot retrieve another customer's order even if they know the `orderID` — the session check will reject it.

### A4. Realtime → Polling Decision

**Why Realtime was removed:**

Supabase Realtime subscriptions require the connecting role to have `SELECT` permission on the subscribed table. The `anon` role deliberately has no `SELECT` on `orders` — granting it would allow any anonymous user to query all orders in the database. There is no way to scope Realtime subscriptions to a session without first granting broad SELECT access.

**What replaced it:**

5-second polling via `fetch_customer_order_by_id(orderId, sessionId)` RPC:

- Each poll is session-scoped — the RPC returns only the customer's own order
- Polling is set up in `QueueConfirmation.tsx` and `MyOrders.tsx`
- Intervals are cleared on component unmount (`clearInterval`) to prevent memory leaks
- If the DB is unreachable, the UI retains the last known state rather than crashing

**Trade-off accepted:** ~5-second delay between staff marking an order complete and the customer seeing the update. Acceptable for a restaurant queue context.

---

### A5. Item Validation at Checkout

Before submitting an order, `Checkout.tsx` filters out any cart items with non-numeric `productID`s:

```ts
const validItems = items.filter(item => !isNaN(item.productId))
```

- Prevents any ghost/placeholder items (e.g. temporarily added items before a DB product ID is assigned) from being sent to the backend
- If `validItems.length === 0` after filtering, order submission is blocked entirely

## Part B — General System Security

### B1. RLS Policies

All policies verified from live database (`pg_policies`).

**`anon` role (customers):**

| Table | Operation | Policy Name |
|---|---|---|
| `products` | SELECT | `anon_select_products` |
| `addons` | SELECT | `anon_select_addons` |
| `guest_sessions` | INSERT | `anon_insert_guest_session` |
| `orders` | INSERT | `anon_insert_orders` |
| `order_items` | INSERT | `anon_insert_order_items` |
| `users` | SELECT | `Anon can read users for login` (pre-existing, for staff login UI) |

`anon` has no SELECT on `orders`, `order_items`, or `guest_sessions`. No UPDATE or DELETE on any table.

**`authenticated` role (staff/admin):**

Full CRUD (`SELECT`, `INSERT`, `UPDATE`, `DELETE`) on: `orders`, `order_items`, `products`, `payments`, `users`.

---

### B2. RPC Functions — Security Types

All functions verified from `information_schema.routines`.

**SECURITY DEFINER** (run with elevated DB privileges, bypass RLS):

| Function | Purpose |
|---|---|
| `verify_guest_session` | Session existence + expiry check (`expires_at > now()`) |
| `place_customer_order` | Rate-limited order creation |
| `fetch_customer_orders` | Session-scoped order list, limited to last 24 hours |
| `fetch_customer_order_by_id` | Session-scoped single order |
| `fetch_customer_order_items` | Session-scoped order items |
| `get_best_sellers` | Order aggregate — DEFINER required since anon has no SELECT on `order_items`. Only counts orders from April 13, 2026 onwards to exclude test/placeholder data. Falls back to `is_best_seller` flag until 10 real orders exist. |
| `cancel_customer_order` | Session-scoped cancellation — verifies session ownership and only cancels if `status = 'Pending'`. Returns `{success: true}` or `{success: false, error: 'not_found'\|'cannot_cancel'}`. |
| `get_my_role` | Returns authenticated user's role (staff/admin use only) |

**SECURITY INVOKER** (run as the calling role, subject to RLS):

| Function | Purpose |
|---|---|
| `generate_queue_number` | Counts today's orders, returns Q-XXX |
| `get_session_id` | Reads `x-session-id` request header |
| `set_queue_number` | Trigger — generates queue number only when `sessionID IS NOT NULL` |

### B3. Frontend Configuration

- **Anon key only** — `client/src/lib/supabase.ts` initializes with `VITE_SUPABASE_ANON_KEY`. The service role key is never used client-side.
- **Env var guard** — startup throws immediately if `VITE_SUPABASE_URL` or `VITE_SUPABASE_ANON_KEY` are missing.
- **Cart is local-only** — cart state lives in `localStorage` and is never synced to the DB between sessions. It only reaches the backend at the moment of order submission.

---

### B4. Error Handling

- `ErrorBoundary.tsx` wraps the entire React app
- Caught errors are logged to `console.error` (developer-visible in browser devtools only)
- Users see only: **"Something went wrong"** + **"Reload Page"** button
- No stack traces, Postgres error messages, or internal details are shown to users
- Per-feature failures (menu load, order fetch) show a retry button with a generic message
