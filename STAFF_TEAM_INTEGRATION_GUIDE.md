# Goat2Go Customer Module — Integration Guide for Staff POS Team

## What This Document Covers

This document explains everything the staff POS and admin dashboard team needs to know about the customer ordering module that now shares the SE2 Supabase database. It covers what was added, how the two systems communicate, what your existing system should do to support customer orders, and action items going forward.

---

## What Is the Customer Module?

A mobile-first web app where restaurant customers can:

1. Scan a QR code at their table to open the ordering page
2. Browse the full menu (loaded live from the `products` and `addons` tables)
3. Select items, choose add-ons, pick Dine-In or Take-Out
4. Apply a discount (PWD or Senior, 20%)
5. Place an order and receive a queue number (e.g. Q-001, Q-002)
6. View their receipt and track order status

The customer module handles no login, no payment processing, and no account creation. It only creates orders. Payment processing and order management remain entirely on the staff side.

---

## How the Two Systems Communicate

The `orders` table is the shared communication channel. Here is the complete data flow:

**Step 1 — Customer places order:**
The customer app creates a row in `orders` with:
- `sessionID` — UUID identifying the customer's anonymous session
- `orderType` — "Dine-In" or "Take-Out"
- `discountType` — "None", "PWD", or "Senior"
- `status` — defaults to "Pending"
- `paymentstatus` — defaults to "Unpaid"
- `queueNumber` — auto-generated by a database trigger (Q-001, Q-002, resets daily)
- `userID` — NULL (no staff user involved yet)

Corresponding rows are created in `order_items` with `productID`, `product_sid`, `quantity`, `price`, and `selectedAddons` (JSONB array of chosen add-ons).

**Step 2 — Staff sees the order:**
Your POS queries `SELECT * FROM orders` and customer orders appear alongside staff-created orders. You can identify customer orders by:
- `sessionID IS NOT NULL` (staff orders have NULL sessionID)
- `queueNumber IS NOT NULL` (staff orders have NULL queueNumber)

**Step 3 — Staff processes the order:**
Your POS updates the order as normal:
- `UPDATE orders SET status = 'Completed' WHERE "orderID" = ...`
- `UPDATE orders SET paymentstatus = 'Paid' WHERE "orderID" = ...`
- Optionally set `userID` to the cashier who processed it
- Optionally set `completeTimestamp`
- Create a `payments` record as usual

**Step 4 — Customer sees the update:**
The customer app polls the database every 5 seconds and picks up the status change automatically.

## What Was Added to the Database

All changes are **additive**. Nothing existing was altered, renamed, or removed. All your existing `authenticated` role policies (19 total) are completely untouched.

### New Tables

| Table | Purpose |
|---|---|
| `guest_sessions` | Anonymous customer sessions. UUID primary key, auto-expires after 24 hours. RLS enabled. |
| `addons` | Menu add-ons customers can select. RLS enabled. |

**Current addons:**

|---|---|---|

| addon-sauce | Extra Sauce | ₱15 |
| addon-no-msg | No MSG | ₱0 |
| addon-spicy | Extra Spicy | ₱0 |

### New Columns on `orders`

| Column | Type | Default | Purpose |
|---|---|---|---|
| `sessionID` | UUID | NULL | Links order to a guest session. NULL for staff-created orders. |
| `queueNumber` | TEXT | NULL | Auto-generated daily (Q-001, Q-002...). NULL for staff-created orders. |
| `discountType` | VARCHAR | 'None' | Customer-selected discount: 'None', 'PWD', or 'Senior'. |

Column defaults also added: `status` defaults to 'Pending', `paymentstatus` defaults to 'Unpaid', `orderTimestamp` defaults to `now()`. The `orderTimestamp` is displayed to customers as full date and time (e.g. "April 13, 2026 · 2:35 PM" on the receipt, "Apr 13, 2026 · 2:35 PM" on the order tracking page).

### New Columns on `order_items`

| Column | Type | Default | Purpose |
|---|---|---|---|
| `selectedAddons` | JSONB | '[]' | Array of selected add-ons, e.g. `[{"id": "addon-rice", "name": "Extra Rice", "price": 30}]` |

### New Columns on `products`

| Column | Type | Default | Purpose |
|---|---|---|---|
| `description` | TEXT | NULL | Product description shown to customers. Pre-populated for all existing products. |
| `is_best_seller` | BOOLEAN | false | Flags products as best sellers on the customer app's homepage. |

### Queue Number Trigger

A trigger `trg_set_queue_number` fires BEFORE INSERT on the `orders` table. It **only generates a queue number when `sessionID IS NOT NULL`**, meaning it only affects customer-placed orders. Staff-created orders (which have no sessionID) are completely unaffected.

Queue numbers reset daily. The first customer order of each day gets Q-001, the second gets Q-002, and so on.

### Realtime

Realtime is enabled on the `orders` table. This allows your POS to use Supabase Realtime subscriptions to detect new customer orders appearing in real time (e.g. for a live order board). The customer app does not use Realtime; it polls via RPC functions instead.

### RPC Functions

Six server-side functions were created for the customer app's use. These run as `SECURITY DEFINER` (elevated privileges) so they can bypass RLS:

| Function | Purpose |
|---|---|
| `place_customer_order(session_id, order_type, discount_type)` | Creates an order and returns orderID, queueNumber, orderTimestamp. Rate-limited to 5 orders per session per 10 minutes. |
| `fetch_customer_orders(session_id)` | Returns all orders for a given session. |
| `fetch_customer_order_by_id(order_id, session_id)` | Returns a single order only if it belongs to the session. |
| `fetch_customer_order_items(order_id, session_id)` | Returns order items only if the parent order belongs to the session. |
| `verify_guest_session(session_id)` | Checks if a session exists. |
| `get_best_sellers(p_limit)` | Returns top N products by total quantity sold. Falls back to `is_best_seller` flag when fewer than 10 orders exist. Excludes add-on products (type = 'Others'). |

These functions are used exclusively by the customer app. Your POS does not need to call them.

### RLS Policies Added (anon role only)

| Table | Policy | Permission |
|---|---|---|
| products | anon_select_products | SELECT (read menu) |
| addons | anon_select_addons | SELECT (read add-ons) |
| guest_sessions | anon_insert_guest_session | INSERT (create session) |
| orders | anon_insert_orders | INSERT (create order) |
| order_items | anon_insert_order_items | INSERT (create order items) |

The `anon` role has **no SELECT access** to orders, order_items, or guest_sessions. All customer reads go through the RPC functions listed above, which verify session ownership internally.

**Your existing `authenticated` role policies are completely untouched.** Full CRUD on all tables for staff.

## What Your POS Needs to Handle

### Displaying Customer Orders

Customer orders show up in the same `orders` table your POS already reads. To distinguish them:

```sql
-- Customer orders only
SELECT * FROM orders WHERE "sessionID" IS NOT NULL ORDER BY "orderTimestamp" DESC;

-- Staff orders only
SELECT * FROM orders WHERE "sessionID" IS NULL;
```

Each customer order has a `queueNumber` (e.g. "Q-001") that the customer sees on their screen. Your POS should display this queue number so staff can call it out when the order is ready.

### Processing Customer Orders

When staff processes a customer order, update the same fields you normally would:

```sql
UPDATE orders
SET status = 'Completed',
    paymentstatus = 'Paid',
    "completeTimestamp" = now(),
    "userID" = <cashier_user_id>
WHERE "orderID" = <order_id>;
```

The customer app polls every 5 seconds and will pick up this status change automatically.

### Reading Selected Add-ons

Customer orders may include add-ons stored in `order_items.selectedAddons` as JSONB:

```json
[
  {"id": "addon-rice", "name": "Extra Rice", "price": 30},
  {"id": "addon-sauce", "name": "Extra Sauce", "price": 15}
]
```

Your POS can display these on the kitchen receipt or order detail screen. If your POS doesn't read this column yet, it's safe to ignore. The column defaults to an empty array `[]` for orders without add-ons.

### Discount Handling

Customer orders include a `discountType` field: "None", "PWD", or "Senior". Both PWD and Senior discounts are 20%. Your POS can use this to:
- Display the discount type on the order
- Compute the discount amount at payment time
- Apply it through your existing payment flow

### 4. Descriptions and Best Sellers (Optional)

The `description` and `is_best_seller` columns on `products` are now live. If your admin dashboard supports editing these fields:

- **`description`**: Text shown under each product on the customer app. Pre-populated for all existing products.
- **`is_best_seller`**: Boolean flag used as a **fallback only**. The customer app now automatically calculates best sellers from real order data (via the `get_best_sellers` RPC). The flag is only used when fewer than 10 real orders have been placed on or after **April 13, 2026** — after that threshold is reached, the top 3 most-ordered products are shown automatically regardless of this flag. Orders before April 13 (test/placeholder data) are intentionally excluded from the calculation. You do not need to manage this manually once real orders start coming in.

  **Note on the home page display:** The home page always shows exactly 3 best sellers. While 10 products are currently flagged, the 3 that appear are simply the flagged products with the lowest `productID` numbers — there is no additional ranking among them. The full list of flagged products is visible on the Menu page under the "Best Sellers" category. Once automatic ranking activates (10+ real orders), the 3 shown will be the top 3 by quantity sold.

The `description` column can be managed from your admin dashboard and reflects on the customer app instantly.

### 5. Add-on Management (Optional)

The `addons` table currently has 4 items. If the restaurant wants to add or modify add-ons, your admin dashboard can manage this table directly. The customer app reads add-ons from the database on every page load.

## Things to NOT Do

- **Do NOT delete or rename** the `sessionID`, `queueNumber`, or `discountType` columns on `orders`. The customer app depends on them.
- **Do NOT delete or rename** the `selectedAddons` column on `order_items`.
- **Do NOT delete** the `guest_sessions` or `addons` tables.
- **Do NOT modify** the RPC functions (`place_customer_order`, `fetch_customer_orders`, `get_best_sellers`, etc.) without coordinating with the customer module developer.
- **Do NOT modify** the `trg_set_queue_number` trigger. It only affects customer orders and is invisible to staff orders.
- **Do NOT drop** the `anon` role policies listed above. They enable the customer app to function.

Your existing tables (`users`, `payments`), columns, and all `authenticated` role policies are completely safe. Nothing the customer module does touches them.
